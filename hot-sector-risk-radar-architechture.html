SECTION 05

数据存储方案

核心决策：行情数据用 Parquet 文件，业务数据用 SQLite。不需要 PostgreSQL / Redis / MongoDB。

5.1 Parquet 文件（行情原始数据 + 计算结果）

为什么 Parquet 更适合行情数据？

• 行情数据只追加不修改 → Parquet 追加模式最合适

• pandas 直读 → DataFrame → 一行代码，无 SQL 转换

• 列式压缩 → 5年全A行情压缩后仅 50-80MB

• 回测时 pandas 直接读 → 比 SQL 查询再转 DataFrame 更快

• 备份 = 复制 data/ 目录

文件结构
data/quotes/a_share/ ├─ 2021.parquet (~12MB) ├─ 2022.parquet ├─ 2023.parquet ├─ 2024.parquet ├─ 2025.parquet └─ 2026.parquet (每天追加) data/quotes/sector/ └─ sector_2021.parquet ... data/quotes/overseas/ ├─ us_index.parquet ├─ forex.parquet └─ commodity.parquet data/computed/ ├─ divergence_2026.parquet ├─ risk_score_2026.parquet └─ market_state_2026.parquet 读取示例： df = pd.read_parquet( "data/quotes/a_share/2024.parquet" )

5.2 SQLite 业务数据

为什么 SQLite 够用？

• 持仓、配置、公告、百分位 → 数据量小（5-10MB），SQLite 零配置

• 需要 CRUD 操作（增删改查）→ SQLite 比 Parquet 更适合

• 单文件，备份 = 复制文件

• 升级到 PostgreSQL 无痛（SQLAlchemy ORM 抽象层）

SECTION 06

多用户设计：专属群模式

核心设计：一个飞书群 = 一个"持仓组"。群内所有人共享持仓、共享风险报告。适合家庭/小团队同管理投资。隔离单位是 chat_id（群ID），不是 open_id（用户ID）。

6.1 群 = 持仓组

🏠 家庭群

夫妻共同管理投资

任何人都能 /add /remove

推送发到群里，都能看到

chat_id: oc_family

📊 研究群

投资讨论小组

共享持仓和风险雷达

推送发到群里

chat_id: oc_research

👤 个人群

自己创建一个只有自己的群

等同私聊模式

完全隐私

chat_id: oc_personal

隔离规则： ├─ 持仓 → 绑定 group_id（群共享，不是个人私有） ├─ 推送 → 发到 chat_id 对应的群（群内所有人可见） ├─ 配置 → 绑定 group_id（每个群可以有不同的推送时间、风险阈值） ├─ 行情/板块/公告 → 公共数据，所有群共享 └─ 操作审计 → 记录 open_id（谁执行的），但数据归属 group_id MVP阶段权限： ├─ 群内所有人平等，都能 add/remove └─ 记录操作者（审计用），但不限制权限 升级阶段权限（可选）： ├─ owner 角色：可以 add/remove，可以改配置 ├─ member 角色：只能查看，不能操作 └─ /role @张三 member → 设置角色

6.2 自动注册流程

触发：机器人被拉入飞书群 ↓ 飞书发送事件：im.chat.member.bot.added ↓ FastAPI /api/feishu/event 接收回调 ↓ event_handler.py 处理： ├─ 提取 chat_id → 检查 groups 表是否已存在 ├─ 不存在 → 自动创建群记录 │ db.insert(Group, feishu_chat_id=chat_id, name=群名, active=True) ├─ 记录拉入机器人的用户为 owner │ db.insert(GroupMember, feishu_open_id=opener_id, role="owner") └─ 在群里发送欢迎卡片： ┌─────────────────────────────────────┐ │ 🛡️ 风险雷达已就位！ │ │ │ │ 接下来在群里发命令即可： │ │ /add 600519 1000 25.5 → 添加持仓 │ │ /help → 全部命令 │ │ │ │ 添加持仓后，系统将开始推送风险雷达 │ └─────────────────────────────────────┘ → 无需手动注册、无需填写表单 → 拉机器人入群 = 自动初始化 → 加了持仓后才开始接收定时推送

6.3 命令执行流程

用户在群内发消息：/add 600519 1000 25.5 ↓ 飞书事件回调 → FastAPI /api/feishu/event ↓ event_handler.py： ├─ 验证飞书签名（防伪造请求） ├─ 提取 chat_id → 查 groups 表 → 获取 group_id ├─ 提取 open_id → 查 group_members 表 → 确认是群成员 │ ├─ 不是成员 → 自动加入（新增 member 记录） │ └─ 是成员 → 继续 ├─ command_parser.py 解析命令： │ → cmd="add", args={"code":"600519","qty":1000,"price":25.5} ├─ actions.py 执行： │ ├─ 验证股票代码（600519是否有效） │ ├─ 查找股票所属板块（是否在板块池内） │ ├─ portfolio_repo.add(group_id, 600519, 1000, 25.5, open_id) │ └─ 记录 command_log ├─ card_builder.py 生成回复卡片 └─ feishu app.send_message(chat_id, card) → 群内所有人都能看到回复：「✅ 已添加持仓 贵州茅台(600519) 1000股」

6.4 定向推送流程

08:00 APScheduler 触发 → push.py 执行 ↓ 1. 查询所有活跃群： groups = db.query(Group).filter(Group.active==True, Group.push_enabled==True) ↓ 2. 生成公共信息（一次计算，所有群共享）： public_data = { market_state: "震荡态", sector_pool: [...], overseas_snapshot: {...}, morning_announcements: [...] } ↓ 3. 对每个群，生成个性化风险报告： for group in groups: ├─ 查询 group 的持仓列表 ├─ 对每只持仓股，查询当日风险评分 ├─ 过滤：风险评分 ≥ group.settings.risk_threshold 的才推送 ├─ 组合 = 公共信息 + 个性化持仓风险 ├─ card_builder.build_daily_card(public_data, personal_data) └─ feishu app.send_message(group.feishu_chat_id, card) ↓ 4. 推送卡片示例： ┌──────────────────────────────────────┐ │ 🛡️ 风险雷达 · 6月3日盘前简报 │ │ │ │ ┌─ 公共部分（所有群相同）──────────┐ │ │ │ 📊 市场状态：震荡态 │ │ │ │ 🔥 热点板块：[AI算力][半导体] │ │ │ │ 💱 美股+0.3% 人民币6.85 │ │ │ └─────────────────────────────────┘ │ │ │ │ ┌─ 个性化部分（本群持仓）──────────┐ │ │ │ ⚠️ 贵州茅台 风险7/10 — 分化触发 │ │ │ │ ✅ 恒瑞医药 风险2/10 — 安心持有 │ │ │ └─────────────────────────────────┘ │ └──────────────────────────────────────┘ 不同群收到不同的个性化部分，公共部分相同 没有持仓的群 → 只收到公共部分，不收到持仓提醒

SECTION 09

关键技术细节

9.1 LLM 严格分工

✅ LLM 只做三件事

1. 公告定性判断（超预期/低于预期/中性）— 数值从爬取中结构化提取

2. 国际事件影响评估（程度判断）— 映射关系硬编码在 sector_mapping.py

3. 风险报告自然语言生成 — 数值从计算引擎获取

❌ LLM 绝不做的事

1. 提取数字（增速50% → 爬取时结构化提取）

2. 计算评分（风险7/10 → 计算引擎算）

3. 计算预期差（+30% → 计算引擎算）

4. 决定影响板块（→ sector_mapping.py 硬编码）

原因：LLM 对数字敏感度不够，必须结构化验证

9.2 采集容错机制

请求 → 限流检查(≥0.5s间隔) → 发送 → 解析 → 清洗 → 写入 │ ├─ 网络异常 → 重试(3次, 指数退避: 1s/2s/4s) ├─ API限流 → 等待后重试 ├─ 数据异常 → 日志+跳过+标记缺失 └─ 批量采集分批执行(每批50只, 间隔3s) 所有采集结果有"数据完整性标记"： ├─ complete → 完整，评分引擎正常使用 ├─ partial → 部分缺失，评分引擎降权使用 └─ failed → 采集失败，评分引擎跳过该日数据

9.3 单体架构启动入口

scheduler/app.py — 系统唯一启动入口 启动时： ├─ 初始化 FastAPI 应用 │ ├─ 注册飞书回调路由（/api/feishu/event, /api/feishu/card） │ ├─ 注册看板API路由（/api/risk, /api/sector等） │ ├─ 注册看板页面路由（/, /detail, /sector等） │ └─ 挂载静态文件（HTML/ECharts/CSS） │ ├─ 初始化 APScheduler │ ├─ 08:00 → job_morning_brief() │ ├─ 10:30 → job_midday_snapshot() │ ├─ 14:00 → job_divergence_check() │ ├─ 15:30 → job_afternoon_summary() │ └─ 20:00 → job_evening_announcement() │ ├─ 初始化飞书应用 │ ├─ 获取 tenant_access_token │ └─ 启动自动刷新（每2小时） │ └─ uvicorn.run(app, host="0.0.0.0", port=8080) → 一个进程包含：FastAPI + APScheduler + 飞书回调 为什么单体？ ├─ 规模小（1-5个群），不需要分布式 ├─ 避免进程间通信的复杂度 ├─ 部署简单：一个Docker容器 └─ 性能瓶颈？→ 5个群的推送 + 采集，单进程绰绰有余

SECTION 12

技术风险

⚠️ AKShare API 不稳定/限流

影响：采集失败导致评分缺失

应对：重试机制(3次指数退避) + 数据完整性标记 + 缺失日跳过评分 + BaoStock作为备用数据源

⚠️ 巨潮/东财网页改版

影响：爬虫失效

应对：爬虫模块独立 + 解析逻辑可快速修改 + 爬虫失败不影响核心功能（分化检测不依赖公告）

⚠️ LLM API 响应慢/超时

影响：推送延迟

应对：数值计算不依赖LLM → 超时则推送纯数值版报告（无LLM解读），LLM解读异步补充

⚠️ SQLite 并发写入锁

影响：多任务同时写入冲突

应对：MVP阶段写入是串行的（定时任务不重叠），升级后迁移PostgreSQL

⚠️ VPS 宕机

影响：漏掉当天推送

应对：Docker restart:always + 健康检查脚本 + 启动时自动补跑当日缺失任务

⚠️ 飞书 API 变更

影响：事件回调格式变化导致命令解析失败

应对：feishu 模块独立 + 事件处理层抽象 + 版本锁定飞书SDK
