## 一、项目背景与目标

### 痛点

日常摄影拍摄后，照片数量多（单次可达数百至上千张），逐张人工筛选耗时费力。需要一个辅助工具先做初筛，用户再对筛选结果做精细化挑选。

### 目标

构建一个**理解摄影意图的照片筛选助手**，而非简单的照片质量评分器。核心价值是减少人工筛选的工作量，而非替代人工判断。

### 定位

- 工具只做**初筛**，不做**终裁**
- 给出建议和理由，最终决定权留给用户
- 边界照片宁可多保留，不要误杀

## 二、核心设计思想

### 2.1 六个基本命题

#### 命题一：评分的本质是偏好模型，不是质量模型

评分的目标不是逼近"照片的客观质量"，而是模拟用户的判断——"你会不会留这张照片"。同一张照片在不同人眼里价值不同，系统应该学会像用户一样判断，而非给出一个"标准答案"。

#### 命题二：维度可组合的本质是让用户定义评估语境

照片质量不是固定维度的——拍人像时"眼睛有没有闭"是致命问题，拍星空时这个维度不存在。可组合不是锦上添花，而是正确的抽象方式。每个维度回答一个特定语境下才有意义的问题，用户通过组合维度来定义评估语境。

#### 命题三：技术评分公式需要可调节的惩罚力度

> **本命题仅适用于技术评估。审美评估使用不同的合成逻辑（见命题六和第三章）。**

简单平均是错的——一张完全模糊的照片不应因为曝光好就得到"还凑合"的综合分。需要一个从宽容到严厉的连续光谱，而非二选一。幂均值（Power Mean）通过参数 p 实现这一点。

#### 命题四：绝对评分与相对评分是两个不同的问题

- 绝对评分回答：这张照片的**留存价值**有多高
- 相对评分回答：在同类照片中，它是不是最佳选择

"这张照片值不值得留"和"这组照片里该留哪张"是两个问题，需要双层评分。

#### 命题五：评分必须可追溯到具体证据，并延伸为后期修复指引

分数本身没有意义，"曝光 65 分，因为左上角高光溢出 15%，肤色区域偏暗 1.3 档"才有意义。可解释性是系统内在属性，不是 UI 层面的事后补充。

更进一步——指出问题只是第一步，自然延伸是告诉用户**怎么修、能不能修、代价多大**。评分应从"问题诊断"升级为"后期可修性评估"。

#### 命题六：审美驱动，技术辅助

传统评分逻辑默认"正确曝光 = 好照片"，但摄影师的真实逻辑是"曝光服务于后期空间"。技术合格的照片可能毫无保留价值（背景杂乱、没有美感），技术有瑕疵的照片可能正是最想留的（完美瞬间但轻微跑焦）。

**技术是底线，审美是决定因素。** 评估架构从"技术参数驱动的质量评分"转向"审美驱动的留存价值评估"，技术从核心降格为辅助。

## 三、评判语境（跨审美与技术的核心概念）

### 3.1 为什么需要评判语境

评分不是在真空中进行的。同一个技术指标，在不同拍摄意图下含义完全不同：

- 欠曝 2 档 + RAW → 后期轻松拉回，不应扣分
- 欠曝 2 档 + JPEG → 信息可能丢失，应扣分
- 欠曝 2 档 + 剪影场景 → 这是创作意图，不是缺陷
- 欠曝 2 档 + 包围曝光帧 → 这是素材，不适用单张标准

### 3.2 评判语境的四层结构

| 语境层 | 要回答的问题 | 影响范围 | 推断方式 |
|--------|-------------|---------|---------|
| **拍摄场景** | 这是什么类型的照片？ | 哪些维度适用、评价标准如何调整 | 场景识别模型（自动） + 预设选择（手动） |
| **后期意图** | 拍摄者是否计划后期？ | 用"当前状态"还是"后期潜力"视角评估 | 文件格式推断（有 RAW = 有后期意图） + 全局开关（手动） |
| **组合归属** | 这张照片是独立作品还是组的一部分？ | 评估单位是单张还是组 | 相似分组 + 包围曝光检测（自动） + 手动分组/解组 |
| **风格偏好** | 拍摄者的审美倾向？ | 什么算"好"（冷调 vs 暖调，锐利 vs 柔和） | 历史筛选行为学习（自动） + 偏好设置（手动） |

### 3.3 评判语境如何影响评估

- **审美评估**：场景决定四个价值维度的权重分布（人像场景情感共鸣权重高，风景场景视觉吸引力权重高）
- **技术评估**：场景 + 后期意图决定评价标准（剪影场景暂停阴影惩罚，RAW 格式切换到后期潜力视角）
- **组合归属**：决定评估单位（包围曝光组不适用单张淘汰逻辑）

**自动推断为主，手动覆盖为辅。** 自动推断不是每次都准，但大部分时候给出合理默认值，用户只需在系统判断错误时纠正。

## 四、审美评估体系（核心层）

### 4.1 设计前提

审美评估不可能给出一个"正确的"分数——审美是主观的、文化的、个人的。设计目标不是精确度量审美，而是**在大部分情况下方向对**，模拟用户的审美判断。

### 4.2 审美判断的本质：先整体，后分解

人的审美判断是"先整体感受，后分析原因"——0.1 秒内产生直觉，然后才去确认技术细节。技术评估是"先分解，后合成"，审美评估顺序相反。

### 4.3 四个价值维度

审美价值分解为四种不同的"值得留下"的理由：

| 维度 | 核心问题 | 例子 |
|------|---------|------|
| **视觉吸引力** | 画面好看吗？ | 光线质感、色彩和谐、构图节奏、背景干净 |
| **情感共鸣** | 有感觉吗？ | 真实表情、氛围情绪、亲密感 |
| **叙事性** | 有故事吗？ | 决定性瞬间、人物互动、环境戏剧性 |
| **稀缺性** | 不可复制吗？ | 转瞬即逝的表情、偶然光线、唯一事件 |

### 4.4 价值维度与测量维度的关系

四个审美价值维度是**高层抽象**——它们回答"值不值得留"。

具体的测量维度（构图质量、色彩和谐度、光线质感、主体突出度等）是**底层指标**——它们为价值维度提供**证据**，但不直接决定留存价值。

```
测量维度（底层指标）                价值维度（高层判断）

  构图质量 ──────┐
  色彩和谐度 ────┤
  光线质感 ──────┼──→ 视觉吸引力
  背景干净度 ────┘

  表情自然度 ────┐
  氛围感 ────────┼──→ 情感共鸣
  亲密感 ────────┘

  瞬间感 ────────┐
  人物互动 ──────┼──→ 叙事性
  环境戏剧性 ────┘

  瞬间唯一性 ────┐
  光线条件 ──────┼──→ 稀缺性
  场景不可复制 ──┘
```

一个测量维度可能同时为多个价值维度提供证据（如"光线质感"既影响视觉吸引力，也影响氛围感/情感共鸣）。

### 4.5 合成逻辑：亮点驱动（OR），不是短板否决（AND）

审美评分和技术评分的合成逻辑完全对偶：

| | 技术评估 | 审美评估 |
|---|---|---|
| 核心逻辑 | 短板否决（AND 倾向） | 亮点驱动（OR 倾向） |
| 合成方向 | 惩罚弱项 | 奖励强项 |
| 糟糕情况 | 有一项致命伤 → 全局否决 | 没有任何亮点 → 全局否决 |
| 好的情况 | 各项都合格 | 有一项突出 |

一张日落不需要同时有故事性——视觉上足够美就行。一张街头可能色彩平庸，但叙事性让它值得留。

### 4.6 输出形式：审美档案（非单一分数）

审美评估不输出一个总分，而是输出一个**审美档案**——核心亮点 + 四维度分布 + 判断依据。一张"视觉 85、其余 30"的照片比"四维都 60"的照片更值得留，因为那一个 85 就是留下它的理由。

### 4.7 整体判断的两种策略

用户可在两种策略间选择，都尝试后决定用哪种模式：

**策略 A：亮点 + 基线双判据**

- 条件 1：max(四个维度) ≥ 亮点阈值（有至少一个留下来的理由）
- 条件 2：min(四个维度) ≥ 基线阈值（没有审美上的致命缺陷）
- 两个条件都满足 → 保留
- 只有条件 1 → 保留但标记"有短板，确认是否接受"
- 只有条件 2 → "技术合格但无亮点"，建议淘汰
- 都不满足 → 淘汰

**策略 B：整体直觉 + 维度验证**

- 第一步：整体直觉判断——这张照片让你想多看一眼吗？（对应人翻照片时 0.1 秒的直觉）
- 第二步：维度归因——直觉是"好"则追问"什么吸引了你？"，直觉是"一般"则追问"缺了什么？"
- 第三步：一致性校验——直觉与维度分解矛盾时标记"请人工确认"

| 组合 | 判定 |
|------|------|
| 直觉"好" + 维度有亮点 | 高置信度保留 |
| 直觉"好" + 维度无亮点 | 不确定，人工审查 |
| 直觉"一般" + 维度有亮点 | 可能被低估，人工审查 |
| 直觉"一般" + 维度无亮点 | 高置信度淘汰 |

## 五、技术评估体系（辅助层）

### 5.1 定位：辅助层，非核心层

技术评估回答两个问题：
1. 这张照片有没有致命的技术硬伤？
2. 如果要留，技术上可不可行？修复代价多大？

### 5.2 维度插件架构

每个评估维度是独立插件，遵循统一接口，可自由组合：

- **元信息**：唯一 ID、名称、分类、说明
- **依赖声明**：依赖的其他维度 ID、激活条件
- **参数定义**：该维度可调参数（阈值、偏好等）
- **统一输出**：0-100 标准化分数 + 置信度 + 分项明细 + 标签

**条件激活**：维度可声明激活条件，不满足时跳过。例如"面部质量"维度只在检测到人脸时激活，无人脸时 confidence=0，自动排除出最终评分。

### 5.3 内置维度

**技术类**：清晰度、曝光、噪点、动态范围、白平衡、色彩准确度

**构图类**：主体突出度、三分法契合度、视觉平衡、引导线构图、地平线水平度

**语义类**：面部质量、人物数量与分布、场景识别、主体检测

> **注意**：色彩和谐度、情感表达力、视觉重心与张力、画面简洁度等审美指标属于审美评估的**测量维度**（见 4.4），不在此列出。它们为审美价值维度提供证据，而非作为技术评估的独立维度参与幂均值合成。

### 5.4 评分管线

> **管线顺序反映"审美先行，技术兜底"原则。**

五阶段管线：

1. **快速硬过滤（Gate Filter）**：纯黑/纯白帧、严重模糊、误触快门、文件损坏 → 不通过直接淘汰，不跑后续阶段
2. **信号提取（Signal Extraction）**：嵌入式 JPEG 预览加载、EXIF 解析、直方图、边缘检测、人脸检测 → 结果缓存供所有维度复用
3. **审美评估（Aesthetic Evaluation）**：整体直觉判断 → 维度归因 → 亮点识别 → 生成审美档案（核心层，决定方向）
4. **技术验证（Technical Validation）**：仅对审美价值不确定或需确认修复可行性的照片执行 → 生成技术档案（辅助层，验证可行性）
5. **综合建议（Recommendation）**：结合审美档案 + 技术档案 → 生成保留/审查/淘汰建议 + 修复指引

### 5.5 分数合成公式：广义加权幂均值（仅用于技术评估）

> **本公式仅适用于技术评估的跨维度合成。审美评估使用亮点驱动（OR）逻辑，不使用此公式。**

```
M_p(s, w) = ( Σ(wᵢ × sᵢᵖ) / Σ(wᵢ) )^(1/p)
```

参数 p 控制严格程度：

| p 值 | 等效 | 行为 |
|------|------|------|
| p = +1 | 算术均值 | 宽容——好项弥补差项 |
| p = 0 | 几何均值 | **推荐默认**——低分显著拖累 |
| p = -1 | 调和均值 | 严格——任何短板不能有 |
| p = -∞ | min(sᵢ) | 极端严格——木桶效应 |

有效权重 = 用户权重 × 置信度。置信度为 0 的维度自动排除。

### 5.6 后期可修性评估与"故意为之"处理

> 后期可修性和"故意为之"是同一框架的两个方面：**评估当前状态与理想状态的偏差，并判断这个偏差是缺陷、可修复问题、还是创作意图。**

#### 偏差分类

| 类型 | 含义 | 处理方式 | 例子 |
|------|------|---------|------|
| **致命伤** | 不可修复 | 直接淘汰 | 完全跑焦、纯黑帧 |
| **可修复伤** | 可后期弥补，有代价 | 保留，标注修复建议和代价 | 轻微欠曝（RAW 可拉回）、白平衡偏移 |
| **无需修复** | 本身就是好的 | 正常保留 | 对焦准确、曝光合理 |
| **故意为之** | 看似缺陷实为意图 | 调整评价标准，不扣分 | 向右曝光、剪影、慢门效果 |

#### 两种评估视角

同一维度提供两个视角的评分：

| 维度 | 当前状态视角 | 后期潜力视角 |
|------|-------------|-------------|
| 曝光 | 直方图偏右，"过曝倾向" | RAW 信息量充足，"后期空间大" |
| 清晰度 | 轻微软焦 | 锐化可补救 |
| 噪点 | 高 ISO 噪点明显 | AI 降噪可处理 |

视角选择由评判语境决定：
- 有后期意图（RAW 格式自动推断）→ 使用"后期潜力"视角
- 无后期意图（仅 JPEG）→ 使用"当前状态"视角
- 筛选阶段用"后期潜力"视角，排序阶段用"当前状态"视角

#### 三种"故意为之"的识别与处理

**第一种：为后期空间服务的"不完美"**（向右曝光 ETTR、宁欠勿曝）
- 识别信号：RAW 格式 + 直方图刻意偏移 + 曝光补偿设置
- 处理：切换到"后期潜力"视角评估

**第二种：创作意图驱动的"不完美"**（慢门流水、剪影、日系过曝）
- 识别信号：场景识别（剪影/长曝光/高调风格）
- 处理：场景理解优先于技术评判——检测到剪影场景则暂停阴影惩罚，改为评估轮廓清晰度；检测到慢门效果则暂停运动模糊惩罚，改为评估拖影美感

**第三种：素材性质的"不完美"**（包围曝光帧、HDR 素材、焦点堆叠）
- 识别信号：EXIF 包围曝光标记 + 相似分组
- 处理：评估单位从"单张"变为"组"，不适用单张淘汰逻辑，改为评估素材齐全度和组内对齐度

### 5.7 配置系统（Profile & Preset）

评分配置结构包含：
- 严格度参数 p（技术评估用）
- 各技术维度的启用/禁用、权重、子参数
- 审美价值维度的权重分布（受评判语境/预设影响）
- 硬过滤阈值
- 相似组设置

内置预设（同时影响审美和技术配置）：人像、风景、街拍、活动/事件、建筑、通用

## 六、审美评估与技术评估的协作

### 6.1 关系定义

- 审美评估（核心层）：决定方向——"值不值得留"
- 技术评估（辅助层）：修正方向——"留不留得住"

### 6.2 协作矩阵

| | 技术OK | 技术有可修复问题 | 技术有致命伤 |
|---|---|---|---|
| **高审美价值** | 保留 ✅ | 保留，标注修复建议 🔧 | 痛苦抉择，显式呈现给用户 💔 |
| **低审美价值** | 淘汰——"技术合格但无亮点" ❌ | 淘汰 ❌❌ | 坚决淘汰 ❌❌❌ |
| **不确定** | 人工审查 ⚠️ | 倾向淘汰 ⚠️ | 倾向淘汰 ⚠️ |

**"痛苦抉择"**（高审美 + 致命技术伤）：不应默默淘汰，而应显式呈现给用户——"这张技术上过不去，但瞬间真的很棒，你来决定"。因为在此边界情况下，用户的个人判断比任何算法都准。

### 6.3 筛选策略顺序

审美先行，技术兜底：

1. 审美初筛——快速判断留存价值方向
2. 技术验证——对不确定的照片做技术评估辅助决策
3. 相似组优选——同组内选审美价值最高的

### 6.4 完整输出示例

一张照片的完整评估结果应包含审美档案 + 技术档案 + 综合建议：

```
照片 IMG_2847 — 综合建议: 保留 ✅

审美档案:
  整体判断: 值得保留
  核心亮点: 情感共鸣 — 自然笑容，聚会氛围温暖
  视觉吸引力: 62 (背景略杂乱)
  情感共鸣: 78 ← 核心亮点
  叙事性: 45
  稀缺性: 70 (表情瞬间不可复制)

技术档案:
  清晰度: 88 ✅
  曝光: 65 ⚠️ 轻微欠曝 1.5 档 → RAW 可拉回，预计噪点增加 2 级
  噪点: 72 ✅ (ISO 1600 可接受)
  白平衡: 80 ✅

修复建议: 拉高曝光 1-1.5 档，注意阴影区域噪点
标签: 人像, 自然光, 微笑, 需轻微曝光修复
```

## 七、相似组与相对评分

### 7.1 相似分组

自动识别连拍/包围曝光等相似照片，归为一组。每组推荐关键帧（基于审美价值最优）。

### 7.2 双重评分

每张照片同时获得：
- **绝对评分**：独立于其他照片的留存价值评估
- **相对评分**（仅组内）：组内排名、与最佳的差距、关键差异说明

### 7.3 组内对比逻辑

- 各维度归一化到组内最高分 = 100
- **审美维度**：仍使用亮点驱动（OR）逻辑——组内选审美亮点最突出的
- **技术维度**：使用幂均值合成——组内选技术综合最好的
- 维度敏感度自动调整：组内方差大的维度（有区分力的）权重提升，方差小的（都差不多的）权重降低

### 7.4 包围曝光识别

检测到包围曝光时，整组标记为"HDR 素材组"，不适用单张淘汰逻辑，改为评估素材齐全度和组内对齐度。

## 八、学习机制

### 8.1 设计目标

学的是"用户与默认审美标准之间的偏差"，而非"什么是好照片"。

### 8.2 学习范围

学习不仅限于维度权重，还包括：
- **审美偏好**：用户更看重哪个价值维度（情感 vs 视觉 vs 叙事 vs 稀缺）
- **技术容忍度**：用户对各类技术瑕疵的容忍程度
- **评判语境偏好**：用户是否偏好某种后期工作流（ETTR？宁欠勿曝？）、是否常拍剪影/慢门等
- **风格偏好**：冷调 vs 暖调、锐利 vs 柔和、干净 vs 氛围感

### 8.3 三层结构

#### 第一层：隐式学习（被动、无感）

记录每次"系统判断 vs 用户实际决定"的偏差，从偏差中提取模式。每积累 10 个有效偏差样本触发权重微调，单次调整幅度 ≤ 0.1，需 30+ 样本稳定，100+ 样本达高置信度。

#### 第二层：主动校准（轻量互动）

- **偏差回顾**：积累偏差后主动呈现摘要，请用户确认推断是否正确
- **边界案例投票**：把系统不确定的照片拿出来让用户快速投票（👍 留 / 👎 不留 / 🤷 随便），比调整抽象权重滑块更有效
- **偏好溯源**：用户可追问"你为什么觉得我偏好 X？"，系统展示导致推断的具体照片和决策记录

#### 第三层：显式偏好声明（场景化配置）

用自然语言和场景化的方式表达偏好，而非数字参数：
- 按场景分开设置（人像/风景/街拍）
- 用选择而非填写
- 同时提供正面（我更在意）和反面（我不喜欢）

### 8.4 三层关系

```
第三层: 显式偏好 ─── 骨架（提供初始方向，加速收敛）
         │
第二层: 主动校准 ─── 肌肉（修正偏差，防止误解积累）
         │
第一层: 隐式学习 ─── 神经（持续微调，越用越准）
```

新用户从第三层开始，使用几次后第一层生效，出现偏差时第二层介入。理想状态：用得越多，第一层比例越高，二三层干预越少。

### 8.5 冷启动机制：理由选择

初期筛选时，删除操作提供理由选项（可选，可跳过）：

**删除理由**：没有亮点平平无奇 / 背景杂乱画面不干净 / 表情姿态不好 / 构图不舒服 / 光线氛围不好看 / 有更好的同类照片了 / 就是感觉不对

**保留理由**：画面好看光线美 / 表情情绪好 / 瞬间难得错过不再有 / 有故事感 / 整体感觉对 / 后期空间大

**设计要点**：
- 选项用大白话说用户心里已有的想法，不是让用户重新分析
- 单选（最重要的那个理由最有学习价值）
- 可跳过（直接点 × 操作，不选理由）
- 保留时不弹理由（同意系统判断 = 无新信息）
- 删除时弹理由（否定系统判断 = 高价值信号）

**渐退机制**：

| 阶段 | 条件 | 行为 |
|------|------|------|
| 冷启动期 | 前 100-200 次操作 | 每次删除/保留都弹出理由选择 |
| 校准期 | 维度权重稳定 + 预测准确率 > 70% | 仅在判断不一致时弹出 |
| 信任期 | 各场景 30+ 样本 + 高置信度 | 几乎不主动问，仅用户主动时出现 |

**理由数据的附加价值**：
- 构建用户的审美画像（你留下/淘汰照片的主要理由分布、风格关键词）
- 长远可训练个人化轻量评估模型

### 8.6 遗忘与重置

- **自然遗忘**：旧样本权重指数衰减（可配置半衰期：3 个月 / 1 年）
- **显式重置**：用户可要求"忘掉之前的偏好"，回归默认基准
- **风格快照**：可保存当前偏好状态为快照（如"2026 年的风格"），可回溯可切换

## 九、整体工作流

### 9.1 阶段一：选择文件夹

```
启动工具 → 选择工作目录 (如 D:\Photo\2026-06-04) → 系统扫描
```

**自动识别**：
- JPG+RAW 配对检测
- 文件格式推断后期意图（有 RAW = 有后期意图）
- 场景推断
- 建议预设（默认选中，可改）

**子文件夹创建**：
- 首次启动时确认："将在此文件夹创建 保留/ 删除/ 待定/ 三个子文件夹"
- 不自动修改文件夹结构，先确认再操作
- 已存在子文件夹时加载已有决策，继续处理未分类照片（断点续传）

### 9.2 阶段二：预处理（后台自动）

| 步骤 | 内容 | 速度 |
|------|------|------|
| 硬过滤 | 纯黑帧、严重模糊、误触快门 | ~30 秒 |
| 信号提取 | EXIF 解析、直方图、人脸检测、场景分类 | ~2 分钟 |
| 相似分组 | 连拍/包围曝光自动识别 | ~1 分钟 |
| 审美初评 | 轻量预判（全量）+ 深度评估（按需） | 渐进式 |
| 技术辅助评估 | 仅对审美不确定区间的照片 | 按需 |
| 生成建议 | 保留/审查/淘汰 | 即时 |

所有步骤后台运行，用户 ~2 分钟后即可开始操作，评估与筛选并行。

### 9.3 阶段三：浏览与筛选（核心人工环节）

**浏览顺序**（先高确信度，后边界）：

1. **相似组**（决策效率最高）——确认推荐帧或换一帧
2. **系统建议淘汰**——快速扫过，确认有没有误杀
3. **系统推荐保留**——确认有没有"其实不想留"的
4. **待审查**——系统不确定的，用户做最终判断

**操作即文件移动**：保留/删除/待定 → 照片（JPG+RAW 配对 + XMP）在子文件夹之间移动（同磁盘为路径重命名，秒级完成）。

**JPG+RAW 配对移动**：操作一张照片时，系统自动同时移动同名的所有关联文件（CR3/NEF/ARW + JPG + XMP），不会出现配对不一致。

**理由选择**：冷启动期删除时弹出（可跳过），按学习置信度渐进消退。

**实时反馈**：显示学习进度、推翻率、与系统判断一致率。

### 9.4 阶段四：收尾

- **误杀检查**：只呈现高风险误杀的照片（审美价值高但技术有瑕疵的），不需要翻遍所有淘汰照片
- **待定清零**：逐张决定待定文件夹中每张的去向
- 最终只剩**保留/**和**删除/**两个文件夹

### 9.5 阶段五：后处理

- **保留/**文件夹 → 带 XMP 导入 Lightroom/Capture One 做后期，评分、标签、修复建议自动可见
- **删除/**文件夹 → 用户确认后手动清空（工具不自动永久删除）
- 学习沉淀——本次决策为下次使用积累偏好

### 9.6 断点续传

- 中途关闭工具后再次打开，自动加载已有决策（保留/删除/待定子文件夹中的照片），只继续处理未分类的照片
- 可选生成 `.session` 隐藏文件记录进度摘要，下次启动时一目了然

## 十、元数据管理

### 10.1 XMP Sidecar

每张照片旁生成同名 .xmp 文件，内容包含：
- 审美档案（整体判断、核心亮点、四维度分数）
- 技术档案（各维度分数、偏差分类、修复建议）
- 标签、星标
- 审查标记

### 10.2 为什么用 XMP

- 跟文件走——照片移到"保留"文件夹，XMP 跟着移，不丢失
- Lightroom/Capture One 能读——评分和标签直接进入后期工作流
- 不修改原文件——XMP 是独立 sidecar，不动 CR3/JPG

### 10.3 JPG+RAW 配对

系统自动识别同文件名不同扩展名的配对关系，作为整体管理。JPG 用于快速预览和审美评估，RAW 存在即触发"后期潜力"评估视角。

## 十一、性能与成本

### 11.1 分级评估策略

不是每张照片都跑完整评估：

- **硬过滤淘汰**：不需要审美评估，纯技术判断
- **明显平庸/明显优秀**：轻量审美预判即可
- **边界地带**：深度审美评估

实际约 60% 用轻量方式，40% 需要深度评估。

### 11.2 相似组只评关键帧

每组只对推荐关键帧做深度评估，其余帧只做与关键帧的差异检测，省去 80% 的组内调用。

### 11.3 预览图代替全尺寸

审美评估使用 JPG 文件（已拍 JPG+RAW），不解全 RAW。绝大多数评估不需要 5000 万像素。

### 11.4 本地优先模型架构

审美评估优先使用本地多模态模型，无需网络，照片不出本机。

| | 本地多模态模型（审美评估） | 本地算法（技术评估） |
|---|---|---|
| 负责 | 审美评估、整体直觉判断、维度归因、组内差异分析 | 硬过滤、信号提取、技术维度评估、相似分组 |
| 成本 | 免费 | 免费 |
| 速度 | 秒级 | 毫秒到秒级 |
| 隐私 | 照片不出本机 | 照片不出本机 |

云端视觉模型作为可选的增强模式（精度更高，需网络和额外成本）。

### 11.5 渐进式呈现

不等全部评估完成——2 分钟后（本地技术评估完成）即可开始处理相似组和明确淘汰，本地审美评估后台并行跑。

### 11.6 综合估算（800 张照片场景，纯本地模式）

| 项目 | 耗时 | 成本 |
|------|------|------|
| 本地技术评估 | ~5 分钟 | $0 |
| 本地审美评估（全量） | ~15-30 分钟（并行） | $0 |
| **用户可开始操作时间** | **~2 分钟** | — |
| **总成本** | — | **$0** |

### 11.7 云端增强模式（可选）

如需更高精度，可配置云端视觉模型对边界照片做深度评估，成本 ~¥1-3/次。

### 11.8 纯本地降级方案（可选）

如本地多模态模型不可用或显存不足，审美评估可降级为本地轻量美学模型（如 LAION Aesthetics Predictor / CLIP-based），功能可用但精度打折，成本 $0。

## 十二、模型选型

### 12.1 已确认配置

| 角色 | 模型 | 部署方式 | 用途 |
|------|------|---------|------|
| 日常推理 | deepseek-v4-flash | 云端 API | 文本推理（不支持图片） |
| 审美评估（主力） | InternVL | 本地（Ollama） | 图片理解、审美判断、维度归因 |
| 审美评估（备用/轻量） | gemma3:4b | 本地（Ollama） | 轻量级图片理解、场景识别、基础审美预判 |

### 12.2 模型分工

```
InternVL（主力）:
  ├─ 深度审美评估（整体直觉 + 维度归因）
  ├─ 组内差异分析
  └─ 审美档案生成

gemma3:4b（备用/轻量）:
  ├─ 快速审美预判（全量扫描，判断"倾向保留/倾向淘汰/不确定"）
  ├─ 场景识别
  └─ InternVL 不可用时的降级方案
```

实际执行策略：
- 全量照片先用 gemma3:4b 做轻量预判（更快，显存占用更少）
- 边界照片和深度评估用 InternVL（更准）
- 两个模型本地运行，照片不出本机，零额外成本

### 12.3 云端增强（可选）

如需更高精度，可额外配置云端视觉模型（Gemini Flash / Qwen-VL / GPT-4o / Claude Sonnet），仅对边界照片调用，成本 ~¥1-3/次。

### 12.4 显存与性能参考

| 模型 | 最低显存 | 审美能力 | 推理速度 |
|------|---------|---------|---------|
| InternVL2-8B | ~12 GB | 较好 | 中等 |
| InternVL2-2B | ~6 GB | 基本可用 | 快 |
| gemma3:4b | ~4 GB | 可用（作为预判） | 快 |
| LAION/CLIP-based | 任意 GPU | 粗略 | 很快 |

## 十三、交互设计

### 13.1 快捷键体系

**原则**：快捷键只给高频决策动作，不常用的鼠标点就行。

| 动作 | 快捷键 | 说明 |
|------|--------|------|
| 保留 | P (Pick) | 最常用，单键 |
| 删除 | X | 最常用，单键 |
| 待定 | U (Unsure) | 不想现在决定 |
| 星标 1-5 | 1 2 3 4 5 | 与 Lightroom 一致 |
| 去除星标 | 0 | 与 Lightroom 一致 |
| 下一张 | → / 空格 | 浏览推进 |
| 上一张 | ← | 回看 |
| 下一组 | Tab | 相似组之间跳转 |
| 放大查看 | Z / F | 按住放大，松开恢复 |

**设计细节**：
- P/X/U 在左手区域，右手可留在鼠标上
- 星标和保留/删除独立：打星标 ≠ 保留，星标写 XMP，保留/删除是文件移动
- 放大是临时的——按住看细节，松开回正常

### 13.2 批量操作

**场景一：相似组批量处理**
- "就选推荐帧，其余删除"——相似组最高频操作，一键完成
- "全部保留"——整组都想要
- "逐张看"——进入组内逐张判断

**场景二：按建议批量处理**
- "一键确认所有建议保留"——信任系统判断的部分批量确认
- 需要用户主动点击确认，工具不自动执行

### 13.3 撤销机制

- 即时撤销：Ctrl+Z 撤销上一步，照片移回原位
- 删除 ≠ 永久消失：照片移到"删除"文件夹，随时可移回
- 只有用户手动清空"删除"文件夹才是真正删除
