# GZH Formatter Skill — 设计决策草稿

> 记录日期：2026-06-19
> 本文件记录了 gzh-formatter skill 的全部设计讨论和决策，供后续调整时回溯参考。

---

## 一、背景与场景

**场景：** 每日由 AI 采集系统收集近 24 小时内的 AI 资讯动态，需要将这些原始数据加工为可直接发布到微信公众号的文章。

**核心约束：**
- 纯资讯速览，不做深度分析
- 内容涉及国内/国外，中英文混杂
- 约 10 条精选
- 定时任务触发（Hermes + Claude Code）

---

## 二、设计决策记录

### 决策 1：内容定位

| 维度 | 结论 |
|------|------|
| 风格 | **纯资讯速览**，每条只陈述事实（1-2 句） |
| 点评 | **不做任何点评/分析**，不写"这意味着什么" |
| 深度观察 | ❌ 不需要，纯简报形式 |
| 价值定位 | 信息密度优先，不是观点输出 |

### 决策 2：板块结构

| 维度 | 结论 |
|------|------|
| 分类方式 | **国内 / 国外 两大板块** |
| 细分分类（大公司/产品/研究等） | ❌ 不需要，保持简单 |
| 板块顺序 | 🇨🇳 国内在前 → 🌏 国际在后 |
| 板块为空 | 不显示空板块标题，不占位 |

### 决策 3：标题格式

| 维度 | 结论 |
|------|------|
| 标题风格 | **头条型**（非固定模板） |
| 格式 | `AI 早报｜{今日头条（中文）}` |
| 头条选择 | 有 score 取最高，无 score 取第 1 条 |
| 头条翻译 | 若头条是英文 → 仅标题部分翻译为中文 |
| 兜底格式 | `AI 早报｜今日 AI 动态速览` |

### 决策 4：输入格式

skill 需要适配以下三种输入格式：

| 格式 | 特征 | 解析方式 |
|------|------|---------|
| **A. 结构化 JSON** | `[{title, url, score, time_str}]` | 直接解析 |
| **B. Google News 文本** | `来源\nMore\n标题\n时间\n作者` | 段落模式解析 |
| **C. 自由粘贴** | 无固定格式 | 启发式行级解析 |

**默认读取路径：** `gzh-formatter-skill/data.json`

### 决策 5：分类策略（国内/国外）

| 优先级 | 判定依据 |
|--------|---------|
| 1 | URL 域名包含 `.cn` |
| 2 | URL 域名包含中文平台（如 36kr、zhihu 等） |
| 3 | 标题包含中文字符 |
| 4 | 兜底 → 国外 |

### 决策 6：排序与精选

| 条件 | 处理 |
|------|------|
| 有 score | 按 score 降序 |
| 无 score | 按原始出现顺序 |
| 总数 > 10 | 精选 10 条，保持国内/国外均衡 |
| 同一来源重复 | 最多保留 2 条 |

### 决策 7：双语处理

| 场景 | 处理 |
|------|------|
| 中文内容 | 正文直接写 1-2 句简讯 |
| 英文内容 | 两段式：中文翻译 + 英文原文 |
| 大标题 | 头条标题必须翻译为中文 |
| 条目标题 | 中文保留原文，英文保留原文 |
| 原文修正 | 不修改原文内容，只做事实提炼 |

### 决策 8：主题系统

| 维度 | 结论 |
|------|------|
| 主题数量 | **7 套** |
| 映射方式 | 星期 → 主题（周一至周日各一套） |
| 输出格式 | inline-style HTML（因公众号需手动粘贴） |
| 深色背景 | ❌ 不支持（微信编辑器强制白底） |
| 周日主题 | 改为白底适配的"深海蓝"配色 |

7 套主题色值详见 `references/writing-style.md`。

### 决策 9：输出格式（v2）

初始版本包含 h1 主标题，实测后去掉——因为主标题与公众号外标题重复，无增量信息。

**替代方案：** 文章正文以**内容速览**模块开头，用 3-4 条概括让读者快速了解当日概貌。

| 维度 | v1（已废弃） | v2（当前采用） |
|------|-------------|--------------|
| 文章开头 | h1 主标题（与外标题重复） | 📋 内容速览（3-4 条主题归纳） |
| 封面图 | 保留 | 保留 |
| 日期 | 保留 | 保留 |

### 决策 10：封面图

**主方案（方案 B）：** skill 直接生成带 inline-style 的 HTML，可粘贴到微信公众号编辑器。

**备选方案（方案 A，已归档）：** 生成结构化 Markdown + 主题标记，由用户在 135 编辑器/秀米中套用模板。

#### 方案对比

| 维度 | 方案 A（135/秀米） | 方案 B（inline HTML） |
|------|-------------------|---------------------|
| 自动化程度 | 低，需手动操作 | 高，一键粘贴 |
| 样式丰富度 | 高，模板可精细设计 | 中等，微信兼容子集 |
| 维护成本 | 需要维护 7 套模板 | 维护 7 套配色即可 |
| 实施难度 | 低 | 低 |
| **选型** | 归档备选 | **✅ 当前采用** |

### 决策 10：封面图

| 维度 | 结论 |
|------|------|
| 图片来源 | 固定使用 `640.jpeg` |
| 存放位置 | `gzh-formatter-skill/640.jpeg` |
| HTML 处理 | 占位 `<img>` 标签，src 待替换 |
| 发布流程 | 用户需将图片上传至微信后替换 HTML 中的 src |

### 决策 11：输出路径与命名

| 维度 | 结论 |
|------|------|
| 输出目录 | `gzh-formatter-skill/output/` |
| 命名规则 | `YYYYMMDD_AI早报.html` |
| 覆盖策略 | 同名文件直接覆盖（日期天然不冲突） |

### 决策 12：Hermes 集成

| 维度 | 结论 |
|------|------|
| Hermes 兼容 | 格式由 Hermes 自行处理适配 |
| SKILL.md 主体 | 面向 Claude Code，保证在 Claude Code 中可执行 |
| 定时触发 | Hermes 的定时任务触发 Claude Code 执行本 skill |

### 决策 13：来源标注

| 维度 | 结论 |
|------|------|
| 标注方式 | 文末统一罗列，正文用 `[N]` 引用 |
| 格式 | `[N] 标题 — 来源名 + URL` |
| 缺少 URL | 标注为 `[N] 未知来源` |

### 决策 14：边界情况

| 情况 | 处理 |
|------|------|
| 数据为空 | 提示无有效数据，不生成 |
| 只有 1 条 | 按标题规则正常处理 |
| 全是国内/国外 | 只显示一个板块 |
| 头条超长（>30 字） | 截断加 `...` |

## 三、最终文件结构

```

├── data.json                   ← 示例数据（JSON 格式）
├── googleNews.txt              ← 示例数据（文本格式）
├── theme-preview.html          ← 7 天主题预览
├── references/
│   ├── title-formulas.md       ← 标题公式
│   └── writing-style.md        ← HTML 模板 + 主题色 + 写作规范
├── output/                     ← 生成输出目录
│   └── YYYYMMDD_AI早报.html
└── DESIGN_DRAFT.md             ← 本文件
```
