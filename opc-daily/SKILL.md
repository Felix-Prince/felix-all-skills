# OPC Daily — 一人公司案例日报

从国内多个平台搜集最新"一人公司（OPC）"案例，筛选 Top 10，整理为结构化日报。

## 核心设计原则

### 反爬安全第一

本 skill 采用**三阶段混合策略**：

```
阶段一：搜索引擎广度扫描（零风控）
阶段二：CDP 浏览器深度挖掘（低频、串行、有间隔）
阶段三：筛选输出
```

**关键约束：**
- CDP 浏览器阶段**不是必须的**——只有在搜索引擎阶段筛选出的候选案例中，某个平台的摘要信息不足以支撑输出质量时，才启动 CDP 深度挖掘
- 每个平台每天最多发起 **3 次** CDP 操作（包括搜索站内和打开链接），超过则回退到搜索引擎已有数据
- 不同平台之间必须串行执行，间隔 **≥60 秒**
- 同一个平台内，连续操作间隔 **≥10 秒**（模拟阅读）
- 当天最多使用 CDP 访问 **3 个不同平台**
- 子 Agent 分治时，子 Agent 之间也必须串行（一个完成后再启动下一个）

**兜底机制：** 如果 CDP 阶段发现异常（被重定向到验证码/登录页、请求超时等），立即停止该平台的所有操作，回退到搜索引擎已有的摘要数据。不重试、不循环。

## 工作流程

### 第一步：确定日期

- 如果用户指定了日期（如"今天""昨天""5月18日"），使用该日期。
- 如果用户说"今天"或没有指定，使用系统的实际日期。
- 如果用户说某个具体日期，以此日期为新鲜度过滤基准。

### 第二步：确定输出模式

| 模式 | 触发条件 | 输出方式 |
|------|---------|---------|
| **本地文档模式**（默认） | 无特殊声明，或用户要求"存到 docs""生成文档" | 写文件 `docs/YYYYMMDD_一人公司案例日报.md` |
| **IM 精简模式** | 用户说"发到飞书""发到 IM""精简版""简单说一下" | 直接返回文本，不写文件 |
| **Agent 模式** | 用户说"给 Hermes/OpenClaw 用""结构化输出""JSON 格式" | 直接返回结构化文本，不写文件 |

判断优先顺序：用户明确指定 > 上下文暗示 > 默认本地文档模式。

### 第三步：阶段一 — 搜索引擎广度扫描（零风控）

**此阶段完全不走浏览器，使用 WebSearch 工具并行发起全部搜索查询。**

**必搜查询（并行发起，不要串行）：**

| # | 查询语句 | 目标平台 |
|---|---------|---------|
| 1 | `一人公司 案例 2026` | 综合 / 知乎 |
| 2 | `OPC 一人公司 创业 案例 site:zhihu.com` | 知乎 |
| 3 | `一人公司 OPC 案例 site:36kr.com` | 36氪 |
| 4 | `一人公司 OPC site:huxiu.com` | 虎嗅 |
| 5 | `一人公司 OPC 2026 创业` | 新华网 / 人民网 / 主流媒体 |
| 6 | `一人公司 OPC 案例 小红书` | 小红书（通过搜索引擎） |
| 7 | `OPC 一人公司 政策 扶持 2026` | 政策新闻 |
| 8 | `"一人公司" "OPC" 融资 案例` | 财经新闻 |

对每条搜索结果，必须提取：
- 人物/公司名称
- 所在城市
- 业务领域
- 关键数据（营收、用户量等）
- 案例亮点简述
- **原文链接**（关键——每个案例必须有可点击的来源链接）

**案例筛选质量准则：**
- 优先选有可验证数据的（营收、用户数、签约合同等）
- 优先选有独特角度的（不同城市、不同行业）
- 避免纯推广软文——找有实质报道内容的
- 越接近目标日期越好

### 第四步：阶段二 — CDP 深度挖掘（可选，受严格安全约束）

#### 4.1 启动前置检查

**只有在以下条件全部满足时，才进入 CDP 阶段：**

1. 搜索引擎阶段的某个候选案例，其平台内容摘要不足以采集到关键数据（人物背景、核心数据、原文全文等）
2. 该平台当天 CDP 操作次数尚未达到上限（≤3 次）
3. 当天已用 CDP 访问的平台尚未超过 3 个
4. web-access 的 check-deps 检查通过（CDP Proxy 可用），且用户确认了安全须知

**如果上述任一条件不满足，跳过此阶段，直接使用搜索引擎已有数据。**

#### 4.2 加载 web-access skill

执行 CDP 操作前，必须加载 web-access skill 并遵循其指引。运行依赖检查：

```bash
node "${HOME}/.hermes/skills/web-access/scripts/check-deps.mjs"
```

**必须向用户显示以下安全须知：**
```
温馨提示：部分站点对浏览器自动化操作检测严格，存在账号封禁风险。已内置防护措施但无法完全避免，Agent 继续操作即视为接受。
```

#### 4.3 串行深度挖掘（严格执行安全约束）

**总频率控制（全局级，不可更改）：**

| 约束 | 数值 | 原因 |
|------|------|------|
| 每平台每日 CDP 操作上限 | **3 次** | 防止单一平台过度访问 |
| 每日涉及的总平台数上限 | **3 个** | 防止跨平台行为关联 |
| 跨平台间隔 | **≥60 秒** | 模拟真实用户切换行为 |
| 同平台连续操作间隔 | **≥10 秒** | 模拟真实阅读/浏览节奏 |
| 单次 tab 生存周期 | **2-5 分钟** | 太快开关 tab 易触发反爬 |

**操作步骤（对每个需要深挖的平台，串行执行）：**

```
平台 A（如小红书）：
  ├─ 创建新后台 tab
  ├─ 等待 3-5 秒（模拟自然加载）
  ├─ /eval 检查是否已登录（检测登录/验证元素）
  │   ├─ 未登录 → 记录"该平台未登录"，跳过此平台，关闭 tab，等待 60 秒
  │   └─ 已登录 → 继续
  ├─ 打开候选案例的完整链接（POST /navigate）
  ├─ 等待 10-15 秒（模拟阅读）
  ├─ 一次性 /eval 提取完整内容（JS 一次遍历 DOM，不要逐条点击提取）
  ├─ 如果需要打开第二个链接，等待 ≥10 秒后再操作
  ├─ 关闭 tab
  ├─ 检测异常：如果过程中被重定向到验证码/登录页等，立即停止此平台的操作
  └─ 等待 60 秒后才进入下一平台

平台 B（如知乎）— 重复上述流程
```

**提取方式（不得逐条点击）：**
```javascript
// 一次性提取页面全部内容 — 适用于文章/笔记详情页
JSON.stringify({
  title: document.title,
  content: document.body.innerText.substring(0, 5000),
  url: window.location.href,
  images: Array.from(document.querySelectorAll('img[src]'))
    .map(i => i.src)
    .filter(s => s.startsWith('http'))
    .slice(0, 5),
  meta: JSON.stringify(
    Object.fromEntries(
      Array.from(
        document.querySelectorAll('meta[name], meta[property]')
      ).map(m => [
        m.getAttribute('name') || m.getAttribute('property'),
        m.getAttribute('content')
      ])
    )
  )
})
```

**异常处理清单（遇到以下任一情况即停止该平台操作）：**
1. 页面被重定向到登录页、验证码页、安全验证页
2. 页面返回 403/429 状态码
3. `/eval` 返回空数据或异常
4. 连续 2 次操作返回相同异常模式
5. 页面渲染时间超过 30 秒

遇到任一异常 → 记录日志（平台+时间+异常类型） → 关闭 tab → 等待 60 秒 → 跳过该平台剩余操作 → 使用搜索引擎已有数据兜底。

#### 4.4 子 Agent 分治规则

如果使用子 Agent 进行深度挖掘，必须遵守以下规则：

1. **子 Agent 之间串行执行**，一个完成后才启动下一个，不要并行
2. 每个子 Agent 的 prompt 中必须写：`必须加载 web-access skill 并遵循其指引`
3. prompt 使用目标导向的描述（"获取这篇帖子的完整正文"），避免用暗示具体手段的动词（"搜索""抓取""爬取"）
4. 每个子 Agent 只负责 **1 个平台**，最多打开 **2 个链接**
5. 子 Agent 数量不得超过 3 个

#### 4.5 内容缓存

当天 CDP 阶段已提取过的 URL，记录到临时缓存（`/tmp/opc-daily-cache.json`），后续不再重复访问。缓存格式：

```json
{
  "date": "2026-05-20",
  "accessed_urls": [
    "https://www.xiaohongshu.com/...",
    "https://www.zhihu.com/..."
  ],
  "platform_counts": {
    "xiaohongshu": 2,
    "zhihu": 1
  }
}
```

### 第五步：阶段三 — 筛选 Top 10

从所有收集到的结果（搜索引擎 + CDP 深度提取）中，按以下标准选出 Top 10：

1. **时效性**——越接近目标日期排名越高
2. **具体度**——有数据指标优于泛泛描述
3. **地域多样性**——覆盖不同城市（无锡、西安、成都、广州、新疆、北京、东莞等）
4. **行业多样性**——覆盖不同赛道（3D打印、AI产品、AI教育、AI情感陪伴、AI Agent、内容营销、文旅、农业等）
5. **新鲜度**——优先选最近几期日报没出现过的案例

目标不是堆"最大"的名字——而是呈现当天 OPC 生态最有代表性的切片。

**去重逻辑：** 如果某个案例在最近 2 期日报中已经出现过，考虑用一个同等质量但没上过榜的替代。

**CDP 数据优先级说明：** 如果一个案例同时有搜索引擎摘要和 CDP 深度提取的数据，优先使用 CDP 提取的完整数据。如果 CDP 提取失败或数据不完整，回退到搜索引擎摘要。

## 设计思路

**三阶段混合架构：** 搜索引擎做广度（零风控快速扫描），CDP 浏览器做深度（低频精准提取），两者互补但不依赖。

**反爬优先于效率：** 宁可拿到不够深的摘要，也不冒账号封禁的风险。所有 CDP 操作都有严格的频率限制、兜底回退、内容缓存。如果当天已用满配额，即使有更好的候选也只用搜索引擎数据。

**内容缓存防重复：** 同一 URL 在同一天内不会访问第二次，避免重复触发平台风控。

**不重试原则：** CDP 阶段遇到异常不重试、不循环。一次失败就回退——重试的代价（触发风控）远大于收益（多拿一点数据）。

**并行搜索引擎 + 串行 CDP：** 搜索引擎用并行查询快速建立候选池（无风险）。CDP 阶段必须串行——不同平台之间间隔 60 秒，模拟真实用户"看完一个站再看另一个"的自然节奏。

**子 Agent 串行分治：** 如果需要用子 Agent 做深度提取，子 Agent 之间也必须串行。并行 CDP 操作容易在浏览器层面留下密集自动化痕迹，触发跨平台行为关联分析。

**当候选超过 10 个时：** 偶尔会有信息量很大的日子。如果找到 15 个以上的强候选，在末尾加一个"荣誉提名"章节，放 2-3 个额外条目（仅姓名 + 一句话）。

**当候选不足 10 个时：** 新闻淡季偶有发生。如实收录，在总结里说"今日仅找到 N 个有效案例"，模板相应缩减。
